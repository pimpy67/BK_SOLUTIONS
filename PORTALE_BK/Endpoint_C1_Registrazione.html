BK Shield — Specifiche API

C1 — Registrazione Cliente

Endpoint: POST /api/client/register
📅 8 giugno 2026 📋 Versione 1.0 👤 Team BK Solutions

Principio architetturale

Il Portale non blocca mai le app in real-time: emette un entitlement firmato (JWT RS256) che dichiara i diritti della licenza (moduli attivi, scadenza, max istanze). La libreria client verifica la firma con la chiave pubblica embedded e applica le regole localmente.

Se la licenza è scaduta o un modulo non è incluso, è la libreria a non sbloccare la funzionalità — il Portale non interviene in real-time.

L'unica chiamata in uscita del Portale è il polling ALARM verso l'ERP interno di BK Solutions, per raccogliere eventi come PAGAMENTO_CONFERMATO o SOSPENDI_LICENZA.

Endpoint

POST /api/client/register

Scopo: Prima iscrizione di una Software House al servizio BK Shield. Il comportamento varia in base alla presenza o meno di una install_key nel body — che distingue la fase trial dalla fase post-acquisto.

Le due fasi di registrazione

Fase 1 — Trial
  • Nessun contratto firmato
  • Nessuna install_key
  • Nessun fingerprint
  • Attivazione immediata (no OTP)
  • Una trial per azienda per prodotto
  • Durata: 30 giorni
  • Identificativo: vat_number + country
Fase 2 — Post-acquisto
  • Contratto firmato, INSTALL_KEY consegnata
  • install_key presente nel body
  • Fingerprint = hash(install_key + hostname)
  • OTP email solo alla prima istanza
  • Istanze successive: automatico
  • Limite: max_istanze da contratto

Request body

// Campi comuni a entrambe le fasi
{
  "product_key":    "string",  // Chiave prodotto inclusa nella libreria client
  "vat_number":     "string",  // P.IVA o VAT number (formato libero, supporto estero)
  "country":        "string",  // Codice paese ISO 3166-1 alpha-2 (es. "IT", "DE", "FR")
  "company_name":   "string",  // Ragione sociale
  "contact_email":  "string",  // Email per notifiche e OTP
  "contact_phone":  "string",  // Telefono — opzionale
  "referent_name":  "string",  // Nome e cognome del referente — opzionale

  // Solo fase post-acquisto (se assenti → flusso trial)
  "install_key":    "string",  // Chiave pre-condivisa generata da BK alla firma contratto
  "fingerprint":    "string"   // hash(install_key + hostname) — identifica l'istanza server
}
Chiave univoca client: la combinazione vat_number + country + product_key identifica univocamente una Software House per un dato prodotto. Il campo country è necessario per supportare clienti con P.IVA estera (DE, FR, ES…).

Note implementative

Generazione license_key

La license_key è un identificatore opaco generato dal Portale con salt random. Formato: LK-[16 caratteri alfanumerici maiuscoli]. La libreria client la salva localmente e la include nell'header di tutte le chiamate successive (Authorization: Bearer <license_key>).

Entitlement JWT RS256

Il payload del JWT contiene:

{
  "sub":          "LK-A1B2C3D4E5F6",   // license_key
  "iss":          "bk-shield",
  "iat":          1749340800,
  "exp":          1752019200,             // = expires_at
  "product_key":  "PROD-XXXX",
  "license_type": "trial",             // trial | subscription
  "modules":      ["modulo_a", "modulo_b"],
  "max_instances": 2                   // solo post-acquisto
}

La firma RS256 usa la chiave privata del Portale. La libreria client ha la chiave pubblica embedded per verifica offline.

Idempotenza

L'endpoint è idempotente: chiamate ripetute con gli stessi parametri restituiscono 200 already_registered con i dati aggiornati, senza duplicare licenze nel DB.

Domande aperte per Alvise: INSTALL_KEY generata manualmente o automaticamente alla firma contratto? Valore di max_istanze come definito contrattualmente? Vedere Checkpoint_Alvise.md §9.
