1

Obiettivi del progetto

BK Shield è un sistema di gestione licenze software B2B sviluppato da BK Solutions. Permette alle Software House clienti di proteggere le proprie applicazioni tramite una libreria client che si registra e valida le licenze presso un server centrale (il Portale).

Modello di business

  • BK Solutions vende licenze enterprise alle Software House
  • La Software House integra la Libreria Client nel proprio server applicativo (non sui dispositivi degli utenti finali)
  • La libreria si auto-registra all'avvio del server e valida periodicamente la licenza
  • Quando la Software House paga BK, l'ERP interno di BK registra il pagamento
  • Il Portale fa polling verso l'ERP interno BK per attivare le licenze
  • Gli utenti finali dell'applicazione della Software House sono fuori dal sistema BK Shield
Glossario: nei documenti precedenti il termine "Fornitore" era ambiguo. In questo documento: Software House = azienda che acquista la licenza da BK e integra la libreria. ERP BK = gestionale interno di BK Solutions (es. Business Central), mai il sistema della Software House.
2

I tre soggetti coinvolti

Libreria Client

Integrata nel server della Software House. Si registra al primo avvio, ottiene l'entitlement firmato, lo salva in cache, invia heartbeat periodici. Verifica la firma offline con chiave pubblica embedded. Parla solo col Portale.

Portale BK-Service

Server centrale di BK Solutions. Riceve le chiamate dalla libreria, emette entitlement firmati, tiene lo stato delle licenze nel DB. Non blocca mai le app in real-time. Unica chiamata in uscita: il polling ALARM verso l'ERP BK.

ERP Interno BK

Gestionale di BK Solutions (es. Business Central). Quando una Software House paga, l'ERP mette l'evento in coda. Espone un endpoint ALARM che il Portale interroga periodicamente. Non è mai il sistema della Software House.

[Libreria Client] ←→ [Portale BK-Service] → [ERP interno BK] nel server della backend centrale gestionale BK Software House es. Business Central ↑ unica chiamata in uscita (ALARM polling)

Principio architetturale e due fasi

Principio fondamentale

Il Portale non blocca mai le app in real-time: emette un entitlement firmato (JWT RS256) che dichiara i diritti della licenza — moduli attivi, scadenza, numero massimo di istanze server. La libreria verifica la firma con la chiave pubblica embedded e applica le regole localmente.

Se la licenza è scaduta o un modulo non è incluso, è la libreria a non sbloccare la funzionalità. Il Portale non interviene in real-time. Questo garantisce il funzionamento anche in modalità offline (server in manutenzione, intranet isolata).

Le due fasi di registrazione

Fase 1 — Trial
  • Nessun contratto firmato
  • Nessuna INSTALL_KEY, nessun fingerprint
  • Attivazione immediata — nessun OTP
  • Identificativo: vat_number + country
  • Una sola trial per azienda per prodotto
  • Durata: 30 giorni, moduli configurati da BK
  • Risposta diretta: license_key + entitlement JWT
Fase 2 — Post-acquisto
  • Contratto firmato, INSTALL_KEY consegnata da BK
  • Fingerprint = hash(install_key + hostname)
  • Prima istanza: OTP email (verifica una-tantum)
  • Istanze successive (fingerprint noto): automatico
  • Limite istanze: max_istanze da contratto
  • Attivazione TRIAL→ACTIVE tramite evento ALARM
  • Anti-frode: blocco se max_istanze superato

Payload entitlement JWT (RS256)

{
  "sub":          "LK-A1B2C3D4E5F6",   // license_key — identificativo univoco
  "iss":          "bk-shield",
  "iat":          1749340800,           // emesso il
  "exp":          1752019200,           // = expires_at licenza
  "product_key":  "PROD-XXXX",
  "license_type": "trial",             // trial | subscription
  "modules":      ["modulo_a", "modulo_b"],
  "max_instances": 2                   // solo post-acquisto
}

Il JWT è firmato con la chiave privata RSA del Portale. La libreria ha la chiave pubblica embedded nel binario e verifica offline senza contattare il Portale.

Endpoint lato CLIENT

Tutte le chiamate sono effettuate dalla Libreria Client (nel server della Software House) verso il Portale. Dopo la registrazione, ogni chiamata include l'header Authorization: Bearer <license_key>.

Riepilogo endpoint

MetodoEndpointScopoFase
POST /api/v1/client/register Registra la Software House, attiva la licenza trial o verifica INSTALL_KEY, emette license_key ed entitlement firmato Trial + Post-acquisto
POST /api/v1/client/register/verify Verifica il codice OTP ricevuto via email (solo prima istanza post-acquisto) Post-acquisto
GET /api/v1/client/license/status Stato corrente della licenza, giorni rimanenti, messaggi in-app da mostrare Entrambe
POST /api/v1/client/heartbeat Telemetria periodica; aggiorna ultimo_heartbeat; anti-clonazione via session_id Entrambe

POST /api/v1/client/license/renew-request Registra l'intenzione di rinnovo. L'attivazione vera arriva poi dall'incasso lato ERP BK tramite ALARM Post-acquisto

C1 — Registrazione

POST /api/v1/client/register

Primo endpoint chiamato dalla libreria all'avvio del server. Registra la Software House e attiva la licenza (trial o post-acquisto) in un unico passo. Restituisce license_key ed entitlement_jwt che la libreria salva in cache.

Request body

{
  "product_key":   "string",  // chiave prodotto inclusa nella libreria
  "vat_number":    "string",  // P.IVA o VAT number (formato libero)
  "country":       "IT",      // codice paese ISO 3166-1 alpha-2
  "company_name":  "string",  // ragione sociale
  "contact_email": "string",  // email per notifiche e OTP
  "contact_phone": "string",  // opzionale
  "referent_name": "string",  // opzionale

  // Solo fase post-acquisto (se assenti → flusso trial)
  "install_key":   "string",  // chiave pre-condivisa da BK al momento del contratto
  "fingerprint":   "string"   // hash(install_key + hostname)
}

Risposte principali

201
trial_activated — Prima registrazione. Restituisce license_key, entitlement_jwt, expires_at, modules[]
200
already_registered — Già registrata, licenza valida. Risposta idempotente con dati aggiornati
202
otp_sent — (Solo post-acquisto) Prima istanza sconosciuta: OTP inviato via email, restituisce pending_token
409
trial_already_used — Trial già scaduta o revocata per questa P.IVA + paese + prodotto
403
max_instances_reached — Nuova istanza sconosciuta ma limite contrattuale già raggiunto
404
product_key_not_found — Chiave prodotto inesistente o disattivata
422
invalid_data — Campi obbligatori mancanti o formato non valido
Dettaglio completo C1: request/response body, diagrammi di sequenza, tabella errori completa → vedere Endpoint_C1_Registrazione.html

C2 — Stato licenza

GET /api/v1/client/license/status

La libreria interroga periodicamente questo endpoint per ottenere lo stato aggiornato della licenza e i messaggi in-app da mostrare all'utente finale (es. avvisi di scadenza).

Response 200

{
  "status":           "ACTIVE",       // TRIAL | ACTIVE | SUSPENDED | EXPIRED
  "license_type":     "subscription",
  "expires_at":       "2027-06-08T23:59:59Z",
  "days_remaining":   365,
  "modules":          ["modulo_a", "modulo_b", "modulo_c"],
  "entitlement_jwt":  "eyJ...",      // entitlement aggiornato da salvare in cache
  "messages": [
    { "type": "info", "text": "Licenza attiva fino al 08/06/2027" }
  ]
}

C3 — Heartbeat

POST /api/v1/client/heartbeat

Chiamata periodica della libreria (es. ogni 30 minuti). Aggiorna ultimo_heartbeat sul dispositivo/istanza nel DB. Il Portale usa i dati per rilevare istanze non autorizzate (anti-frode via session_id).

Request body

{
  "session_id":   "uuid-generato-all-avvio",  // ID runtime univoco per questa esecuzione
  "fingerprint":  "string",                    // hash(install_key + hostname)
  "timestamp":    "2026-06-08T10:00:00Z"
}

Response 200

{
  "ack": true,
  "entitlement_jwt": "eyJ..."   // eventuale entitlement aggiornato (es. dopo PAGAMENTO_CONFERMATO)
}
Anti-frode: se lo stesso fingerprint invia heartbeat con session_id diversi contemporaneamente, il Portale rileva una potenziale clonazione e segnala l'anomalia (Workflow E).

C4 — Refresh entitlement

POST /api/v1/client/license/refresh

Rinnova il documento JWT firmato prima che scada la finestra di grazia offline. Chiamato proattivamente dalla libreria quando l'entitlement in cache è prossimo alla scadenza e il server ha connettività.

Response 200

{
  "entitlement_jwt": "eyJ...",    // nuovo JWT con nuova exp
  "expires_at":      "2026-06-15T23:59:59Z"
}

C5 — Richiesta rinnovo

POST /api/v1/client/license/renew-request

Registra l'intenzione di rinnovo della licenza da parte della Software House. Non attiva il rinnovo: l'attivazione effettiva avviene quando il Portale riceve l'evento PAGAMENTO_CONFERMATO dall'ERP BK tramite ALARM.

Response 200

{
  "status":   "renew_request_registered",
  "message":  "Richiesta registrata. Il rinnovo sarà attivato alla conferma del pagamento."
}
5

ALARM — Unica chiamata in uscita

Il Portale non riceve push dall'ERP BK: fa polling periodico (pull). Questa è l'unica chiamata in uscita del Portale verso sistemi esterni. L'ERP BK non chiama mai il Portale.

GET {erp_base_url}/alarm?codice={product_code}

Poll periodico configurabile. L'ERP BK restituisce gli eventi pendenti per il prodotto specificato. Il Portale li elabora e conferma con un ack per marcarli come processati.

Response ERP → Portale

{
  "eventi": [
    {
      "id_evento_erp":  "ERP-20260608-001",
      "tipo_evento":    "PAGAMENTO_CONFERMATO",
      "payload": {
        "vat_number":  "IT01234567890",
        "country":     "IT",
        "product_key": "PROD-XXXX",
        "importo":     1200.00,
        "rif_fattura": "FT-2026-0042"
      },
      "timestamp": "2026-06-08T09:30:00Z"
    }
  ]
}

Flusso elaborazione

Portale ERP interno BK │ │ │ GET /alarm?codice=PROD-XXXX │ │ ─────────────────────────────> │ │ { eventi: [...] } │ │ <───────────────────────────── │ │ │ │ Elabora ogni evento │ │ Aggiorna licenze nel DB │ │ │ │ POST /alarm/ack │ │ { id_evento_erp: "..." } │ │ ─────────────────────────────> │ │ 200 OK │ │ <───────────────────────────── │

Eventi trasportati dall'ALARM

PAGAMENTO_CONFERMATO
▶ Trigger attivazione

Pagamento ricevuto da BK. Il Portale attiva la licenza: TRIAL → ACTIVE o rinnova la scadenza. Idempotente via rif_erp.

NEW_CUSTOMER
ℹ Info

Nuova Software House registrata nel gestionale BK. Il Portale può pre-creare il record clienti.

NEW_INVOICE
ℹ Info

Nuova fattura emessa. Record fiscale utile per audit e riconciliazione.

NEW_PRODUCT
ℹ Info

Nuovo prodotto aggiunto nel catalogo BK. Il Portale aggiorna la tabella Prodotti.

CMD_SOSPENDI
⚠ Comando

Sospende la licenza. La libreria riceverà al prossimo heartbeat/refresh un entitlement con stato SUSPENDED.

CMD_RIATTIVA
⚠ Comando

Riattiva una licenza sospesa. Stato → ACTIVE, nuovo entitlement emesso.

CMD_REVOCA
⚠ Comando

Revoca definitiva della licenza. Stato → REVOKED. La libreria smette di sbloccare le funzionalità.

CMD_PROROGA
⚠ Comando

Estende la data di scadenza senza un nuovo pagamento (es. grazia commerciale).

CMD_MODIFICA_LIMITI
⚠ Comando

Aggiorna max_istanze o moduli abilitati su una licenza attiva.

Sicurezza

JWT RS256 — Firma asimmetrica

Il Portale firma l'entitlement con la chiave privata RSA. La libreria verifica con la chiave pubblica embedded nel binario. Nessuna chiamata di rete necessaria per la verifica → funziona offline.

INSTALL_KEY — Post-acquisto

Chiave pre-condivisa generata da BK alla firma del contratto. Inserita nel config del server della Software House. Prova che il server ha il diritto contrattuale di operare. Mai trasmessa in chiaro su rete non sicura.

Fingerprint — Identità istanza

Solo nel post-acquisto: hash(install_key + hostname). Identifica ogni istanza server in modo stabile (anche su VM/Docker/cloud). Salvato in Dispositivi e incastonato nell'entitlement firmato.

Approccio: firma, non cifratura

L'entitlement è firmato ma non cifrato: il contenuto è leggibile, ma non modificabile senza invalidare la firma. Questo è sufficiente per garantire l'integrità dei diritti dichiarati.

La cifratura AES aggiuntiva è in valutazione (domanda aperta per Alvise — vedere Checkpoint_Alvise.md §7).

Anti-frode (Workflow E)

ScenarioRilevamentoAzione
Copia INSTALL_KEY su server non autorizzato Fingerprint sconosciuto + max_istanze raggiunto 403 max_instances_reached + email alert al referente
Stesso server clonato (VM snapshot) Due heartbeat simultanei con stesso fingerprint, session_id diversi Anomalia segnalata, entitlement sospeso in attesa verifica
INSTALL_KEY revocata Portale riceve CMD_REVOCA dall'ERP BK 403 install_key_revoked a tutti i fingerprint associati

Transizioni

TRIAL → ACTIVE ALARM: PAGAMENTO_CONFERMATO — la Software House ha pagato BK
ACTIVE → SUSPENDED ALARM: CMD_SOSPENDI — mancato pagamento rinnovo o comando manuale BK
SUSPENDED → ACTIVE ALARM: CMD_RIATTIVA o PAGAMENTO_CONFERMATO successivo
ACTIVE → EXPIRED Cron job notturno: scadenza superata senza rinnovo confermato
ACTIVE → ACTIVE ALARM: PAGAMENTO_CONFERMATO rinnovo — aggiorna data_scadenza e firma nuovo entitlement
* → REVOKED ALARM: CMD_REVOCA — revoca definitiva, nessun rientro possibile

Licenza provvisoria

Alla scadenza di una licenza ACTIVE, il Portale può creare automaticamente una licenza provvisoria: stessi moduli, durata ~30 giorni, in attesa del pagamento del rinnovo. Se entro 30 giorni non arriva PAGAMENTO_CONFERMATO, la licenza passa a SUSPENDED. (Comportamento esatto da confermare con Alvise — Checkpoint §2 e §6)
