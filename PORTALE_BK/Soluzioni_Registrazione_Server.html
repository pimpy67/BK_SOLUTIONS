Certificazione Server — 4 Soluzioni a Confronto

Progetto: BK Invoice Service  |  Data: 8 giugno 2026  |  Argomento: Come certifichiamo il server del Fornitore alla registrazione?

Glossario dei ruoli
TermineChi èNel sistema
BK SolutionsSviluppa e vende il servizio di licensingOperatore del Portale e dell'ERP
Software HouseAzienda che acquista la licenza da BK e integra la libreria nel suo serverdbo.Clienti nel DB
ERP BKGestionale interno di BK Solutions (es. Business Central)Genera gli eventi ALARM — non è un sistema della Software House
Utente finaleChi usa l'app della Software HouseFuori dal nostro sistema

⚠️ Nota: Il Portale fa polling ALARM verso l'ERP interno di BK Solutions, non verso sistemi della Software House. Quando la Software House paga BK, è l'ERP di BK a registrare il pagamento e a generare l'evento di attivazione.

Contesto: La libreria BK è installata sul server della Software House (l'azienda cliente di BK), non sul dispositivo dell'utente finale. Esistono due momenti distinti in cui avviene la registrazione:
  • Trial — la Software House è sconosciuta a BK, nessun contratto, nessuna chiave pre-condivisa
  • Post-acquisto — il contratto esiste, BK conosce già la Software House
Le soluzioni 1 e 3 coprono questi due scenari diversi. I tre rischi da coprire in entrambi:
  • ✅ Server legittimo della Software House
  • ❌ Server di qualcuno che ha rubato le credenziali
  • ❌ Server aggiuntivo oltre il limite di istanze acquistate

Soluzione 1  Email + Codice OTP  Scenario: TRIAL

La Software House non ha ancora firmato nulla con BK. Vuole provare il prodotto gratuitamente per 30 giorni. Non esiste nessuna chiave pre-condivisa: l'OTP via email è l'unica verifica di identità possibile.

Da dove prendiamo l'email per mandare l'OTP?

La Software House è sconosciuta al Portale. Tre sotto-opzioni:

1a — La Software House include l'email nella chiamata di registrazione (più semplice) 
POST /client/register
{
  "p_iva":           "12345678901",
  "ragione_sociale": "ABC SRL",
  "email":           "admin@abcsrl.it",   ← fornita dal Fornitore
  "product_key":     "BK-INV-001",
  "fingerprint":     "a3f7c2d1..."
}
Il Portale manda l'OTP a quell'email. Semplice, ma la Software House potrebbe inserire un'email non sua.
1b — L'email viene ricavata dalla validazione P.IVA (servizio esterno) Se il servizio di validazione (es. VeriFactura) restituisce anche i contatti ufficiali dell'azienda, usiamo quell'email. Più sicuro, ma dipende da cosa il servizio espone — da chiedere ad Alvise.
1c — La Software House si registra prima sul portale web BK, poi installa la libreria 
Portale web BK:
  [Form registrazione trial] → Fornitore inserisce P.IVA, email, ragione sociale
  → Verifica email → Account creato → BK genera TRIAL_KEY
  → Fornitore riceve TRIAL_KEY via email

Libreria sul server:
  POST /client/register { p_iva, product_key, fingerprint, trial_key }
  ← 200 OK + registration_token  (nessun OTP aggiuntivo)
La verifica umana avviene una sola volta sul web, poi il server funziona in autonomia. È il modello usato da Stripe, Twilio, SendGrid: prima crei l'account, poi usi la chiave.
Semplificazione: nella TRIAL non serve il fingerprint. La P.IVA è già univoca per azienda — basta verificare che non esista già una trial per quella P.IVA + product_key. Il fingerprint entra in gioco solo nel post-acquisto, per contare le istanze server.

Soluzione 2  INSTALL_KEY pre-condivisa  Scenario: POST-ACQUISTO

La Software House ha già firmato il contratto con BK. BK genera una chiave univoca e la consegna alla Software House. Il server si registra automaticamente senza alcun intervento umano.

[Alla firma del contratto]
BK genera INSTALL_KEY univoca → consegnata alla Software House via portale/email sicura
Software House configura il server: INSTALL_KEY=BK-a3f7-c2d1-e9b4-xxxx

[Ad ogni avvio del server]
Server → POST /register { p_iva, product_key, fingerprint, install_key }
       ← 200 OK + registration_token  (nessuna interazione umana)
VantaggiSvantaggi
✅ Completamente automatico, supporta CI/CD
✅ Nessun intervento umano necessario
✅ Standard enterprise (come le API key) 		❌ Se la chiave viene rubata, chiunque può usarla
❌ Nessuna verifica che il server appartenga davvero alla Software House
❌ Richiede processo sicuro di consegna iniziale

Adatta per: Software House con infrastrutture automatizzate e deploy frequenti.

Soluzione 3 ⭐ Consigliata  INSTALL_KEY + OTP solo alla prima istanza  Scenario: POST-ACQUISTO

La Software House ha già firmato il contratto con BK. BK conosce la sua P.IVA, email e quante istanze ha acquistato. La verifica via OTP avviene una sola volta per ogni nuova istanza server; i riavvii successivi sono completamente automatici.

Nota importante: questa soluzione si applica dopo la trial (o per acquisti diretti senza trial). Per la prima registrazione di una Software House sconosciuta si usa la Soluzione 1.
Qui compare il fingerprint per la prima volta. Nel post-acquisto la Software House può avere più istanze server (prod, staging, dev). Il fingerprint = hash(INSTALL_KEY + hostname) identifica ogni singola istanza e permette di:
  • Contare le istanze attive vs max_istanze contrattuale
  • Riconoscere un riavvio dello stesso server (nessun OTP)
  • Rilevare tentativi di copia non autorizzata (Workflow E)

Soluzione 4  Link email  Variante della Soluzione 1

Anziché un codice OTP numerico, l'email contiene un link cliccabile. L'admin apre il link nel browser e la registrazione si sblocca. Il server fa polling per sapere quando il link è stato cliccato.

Server → POST /register { p_iva, product_key, fingerprint }
       ← 202 Accepted + pending_token
       → Email: "Clicca qui per attivare: https://portale.bk/activate?token=eyJ..."
Admin  → Clicca il link nel browser → Pagina di conferma BK
Server → GET /register/status?pending_token=...  (ogni 5 sec)
       ← 200 OK + registration_token  (dopo che il link è stato cliccato)
VantaggiSvantaggi
✅ Più semplice per l'admin (un clic, nessun codice)
✅ Link con scadenza automatica (es. 15 minuti)
✅ Nessun rischio di errore di trascrizione 		❌ Richiede accesso a un browser da parte dell'admin
❌ Non funziona in ambienti headless o intranet isolata
❌ Il server deve fare polling per sapere quando confermato

Adatta per: Fornitori con admin non tecnici che preferiscono un'interfaccia web.

Domande per Alvise

  1. La Software House usa CI/CD e deploy automatici? → Se sì, Soluzione 2 o 3 è obbligatoria
  2. Per la trial: la Software House include l'email nella chiamata (opzione 1a) o si registra prima sul portale web BK (opzione 1c)?
  3. Preferiamo OTP numerico (Sol. 1 e 3) o link email (Sol. 4)?
  4. La INSTALL_KEY viene generata manualmente da un operatore BK o automaticamente alla firma del contratto?
  5. In caso di istanza non autorizzata: blocchiamo subito o mandiamo prima un alert alla Software House?
